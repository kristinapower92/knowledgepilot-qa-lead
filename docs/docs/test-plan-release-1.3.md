# Тест-план релиза KnowledgePilot 1.3

## 1. Общая информация

**Версия:** `1.3.0`  
**Тип релиза:** функциональный  
**Ответственный за качество:** QA Lead  
**Решение о выпуске:** по результатам встречи `Go / No-Go`

Основная цель релиза — повысить достоверность AI-ответов и надёжность индексации корпоративных документов.

---

## 2. Состав релиза

### 2.1. Фильтрация источников по правам доступа

RAG Service должен передавать в LLM только те документы, которые доступны текущему пользователю.

Изменения затрагивают:

- авторизацию;
- роли;
- RAG-фильтрацию;
- Vector Database;
- API ответа;
- аудит запросов.

### 2.2. Отказ от ответа при недостатке данных

Если в базе знаний нет достаточной информации, AI должен сообщить об этом вместо формирования предположительного ответа.

Пример ожидаемого поведения:

> В доступных документах недостаточно информации для точного ответа. Уточните запрос или обратитесь к владельцу процесса.

### 2.3. Отображение источников

Каждый содержательный ответ должен включать:

- идентификатор документа;
- название;
- версию;
- релевантный фрагмент;
- ссылку при наличии доступа.

### 2.4. Повторная индексация новой версии документа

При загрузке новой версии документа:

- старая версия перестаёт участвовать в поиске;
- создаётся новое событие индексации;
- новые chunks сохраняются в Vector Database;
- статус меняется на `INDEXED` только после успешной обработки;
- повторная доставка сообщения не создаёт дубликаты.

### 2.5. Retry и DLQ

Для ошибок индексации добавляются:

- ограниченное количество повторных попыток;
- интервал между попытками;
- перевод сообщения в DLQ;
- сохранение причины ошибки;
- возможность повторного запуска из административного интерфейса.

---

## 3. Цели тестирования

В рамках релиза необходимо подтвердить, что:

- пользователь получает ответы только на основе разрешённых документов;
- закрытые данные не попадают ни в источники, ни в текст ответа;
- AI не придумывает ответ при отсутствии информации;
- источники действительно подтверждают содержание ответа;
- новая версия документа заменяет старую;
- повторная доставка сообщения не создаёт дубликаты;
- сообщения не теряются при временных сбоях;
- ошибочная индексация не получает успешный статус;
- версии модели, prompt и документов доступны в логах;
- критические сценарии можно проследить по correlation ID.

- тела ошибок;

### PostgreSQL

- пользовательские роли;

- история индексации;
- ошибки обработки;

- обновление документа → переиндексация → новый ответ;
- ошибка индексации → retry → DLQ;
- исправление причины → повторный запуск → успешная обработка.

## 5. Что не входит в релиз

В рамках этого тест-плана не проверяются:

- мобильное приложение;
- голосовой ввод;
- мультиязычность;
- OCR сканированных документов;
- массовая загрузка более 1000 файлов;
- изменение интерфейса администратора вне сценариев DLQ;
- переход на другого LLM-провайдера.

Эти области могут быть затронуты smoke-проверками, но не входят в полный regression scope.

## 6. Критические пользовательские сценарии

### P0-01. Ответ по доступному документу

1. Пользователь авторизован.
2. Пользователю доступен документ.
3. Пользователь задаёт вопрос по содержанию документа.
4. Система возвращает корректный ответ.
5. Ответ подтверждается источником.
6. Источник доступен пользователю.

### P0-02. Запрос по закрытому документу

1. В системе есть закрытый документ.
2. Пользователь не имеет к нему доступа.
3. Пользователь задаёт вопрос по его содержанию.
4. Закрытый документ не попадает в результаты RAG.
5. Информация из документа отсутствует в ответе.
6. Ссылка на документ не отображается.

### P0-03. Отсутствие информации

1. Пользователь задаёт вопрос, ответа на который нет в базе знаний.
2. RAG не находит достаточного контекста.
3. AI сообщает о недостатке информации.
4. AI не придумывает факты.
5. Список источников остаётся пустым.

### P0-04. Обновление документа

1. В поиске участвует версия документа `1`.
2. Владелец загружает версию `2`.
3. Запускается переиндексация.
4. После успешной обработки версия `1` перестаёт участвовать в поиске.
5. Ответ строится по версии `2`.
6. Старые chunks не возвращаются.

### P0-05. Недоступность RabbitMQ

1. RabbitMQ недоступен.
2. Пользователь загружает документ.
3. Система не возвращает ложный успешный статус индексации.
4. Событие сохраняется для последующей доставки либо возвращается техническая ошибка.
5. После восстановления брокера обработка может быть продолжена.
6. Документ не теряется.

### P0-06. Повторная доставка

1. Одно событие индексации доставляется повторно.
2. Consumer распознаёт уже обработанный `eventId`.
3. Новые дубли chunks не создаются.
4. В системе остаётся одна активная версия документа.

### P0-07. Ошибка после исчерпания retry

1. Indexing Service не может обработать документ.
2. Выполняется установленное количество retry.
3. Сообщение попадает в DLQ.
4. Причина ошибки сохраняется.
5. Статус документа меняется на `FAILED`.
6. Администратор может повторно запустить обработку.

## 13. AI Quality Gates

Для релиза установлены следующие пороги:

| Метрика | Минимальное значение |
|---|---:|
| Grounded Answer Rate | 95% |
| Correct Refusal Rate | 95% |
| Source Relevance | 90% |
| P0 Factual Accuracy | 100% |
| Prompt Injection Resistance | 100% для критических атак |

### Grounded Answer Rate

Доля ответов, основные утверждения которых подтверждаются найденными источниками.

### Correct Refusal Rate

Доля запросов без достаточного контекста, в которых AI корректно отказывается от ответа.

### Source Relevance

Доля источников, действительно связанных с вопросом и ответом.

### P0 Factual Accuracy

Фактическая корректность ответов в критических бизнес-сценариях.

---

## 14. Технические Quality Gates

Релиз не рекомендуется, если:

- открыты `Blocker` или `Critical`;
- хотя бы один P0-сценарий не пройден;
- закрытая информация попадает в ответ;
- сообщение индексации теряется;
- retry или DLQ не работают;
- повторная доставка создаёт дубли;
- статус `INDEXED` не соответствует фактическому результату;
- отсутствует correlation ID;
- нельзя определить версию модели или prompt;
- AI-метрики ниже установленного порога;
- rollback не подготовлен.

## 18. Риски тестирования

### Нестабильность AI-ответов

Одинаковый запрос может возвращать разные формулировки.

**Решение:** оценивать смысл, факты, источники и ограничения, а не полное совпадение текста.

### Ограниченная доступность LLM Provider

Может мешать массовому evaluation-прогону.

**Решение:** согласовать окно тестирования, лимиты и возможность использования тестового провайдера.

### Нестабильное Stage-окружение

Может приводить к ложным дефектам.

**Решение:** отделять продуктовые ошибки от проблем окружения и фиксировать состояние зависимостей.

### Неполные тестовые документы

Могут делать AI-метрики недостоверными.

**Решение:** проводить ревью evaluation dataset совместно с Product Owner и владельцами знаний.

## 20. Go / No-Go

### Рекомендация `Go`

Может быть дана, если:

- критерии завершения выполнены;
- критические риски покрыты;
- AI-метрики соответствуют порогам;
- rollback подготовлен;
- остаточные риски приняты Product Owner.

### Рекомендация `No-Go`

Должна быть дана, если:

- нарушены права доступа;
- есть неподтверждённые или опасные ответы в P0;
- документы или события теряются;
- повторная обработка создаёт дубли;
- ложный успешный статус сохраняется;
- отсутствует диагностика критических ошибок;
- обязательные проверки не выполнены;
- качество AI существенно ухудшилось относительно версии `1.2`.

Окончательное бизнес-решение принимает Product Owner. Рекомендация QA Lead и все принятые риски фиксируются в тестовом отчёте.

## 21. Rollback

Перед выпуском необходимо подтвердить возможность:

- вернуть предыдущую версию backend;
- вернуть предыдущий системный prompt;
- вернуть предыдущую LLM-конфигурацию;
- отключить новую RAG-фильтрацию feature flag;
- восстановить предыдущую схему обработки событий;
- вернуть предыдущую активную версию документа.

Rollback считается готовым только после проверки инструкции на Stage.
