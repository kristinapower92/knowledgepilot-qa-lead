# Матрица рисков KnowledgePilot

## 1. Цель

Матрица рисков помогает определить:

- какие части продукта проверять в первую очередь;
- какие сценарии блокируют релиз;
- где требуется автоматизация;
- какие риски нужно контролировать после выхода в production;
- кто отвечает за снижение каждого риска.

Тестирование планируется не от количества требований, а от возможного ущерба для пользователя и бизнеса.

---

## 2. Метод оценки

Для каждого риска оцениваются:

### Вероятность

| Балл | Описание |
|---:|---|
| 1 | Маловероятно |
| 2 | Возможно в редких условиях |
| 3 | Может происходить периодически |
| 4 | Высокая вероятность |
| 5 | Почти неизбежно без дополнительных мер |

### Влияние

| Балл | Описание |
|---:|---|
| 1 | Незначительное неудобство |
| 2 | Ограниченное влияние на часть пользователей |
| 3 | Нарушение важного сценария |
| 4 | Серьёзный ущерб продукту или бизнесу |
| 5 | Утечка данных, опасное решение или потеря критичной информации |

4. Меры по снижению рисков
R-01. Доступ к закрытым документам

Меры:

фильтрация документов по ролям до передачи данных в LLM;
негативные проверки доступа через API;
тестирование смены ролей и инвалидации кэша;
проверка отсутствия закрытой информации не только в ссылках, но и в тексте ответа;
аудит обращений к документам.

Релиз блокируется: да.

R-02. Фактически неверный AI-ответ

Меры:

evaluation dataset с эталонными фактами;
проверка ответа по содержанию источников;
ручное ревью критичных сценариев;
автоматическая проверка обязательных фактов;
ограничение формулировок при недостаточной уверенности;
мониторинг жалоб и негативной обратной связи.

Релиз блокируется: да.

R-03. Ответ без подтверждения источниками

Меры:

обязательное поле sources в API-контракте;
проверка релевантности каждого источника;
запрет на успешный ответ без подходящего контекста;
отдельная метрика grounded answer rate;
регрессия ссылок после обновления документов.

Релиз блокируется: да для критических сценариев.

R-04. Галлюцинация при отсутствии данных

Меры:

набор вопросов, на которые в базе знаний нет ответа;
ожидаемая стратегия отказа;
проверка разных формулировок одного запроса;
проверка поведения при пустом результате RAG;
запрет на дополнение ответа знаниями вне корпоративной базы для закрытых сценариев.

Релиз блокируется: да.

R-05. Prompt injection

Меры:

отдельный security-набор запросов;
попытки раскрытия системного prompt;
инструкции игнорировать предыдущие правила;
вредоносные инструкции внутри загруженных документов;
разграничение пользовательского текста и системных инструкций;
регулярное обновление набора атак.

Релиз блокируется: да при обходе прав доступа или системных ограничений.

R-07 и R-08. Неактуальные версии документов

Меры:

версионирование документа;
проверка удаления или деактивации старых chunks;
сравнение поисковой выдачи до и после переиндексации;
контроль статусов в PostgreSQL и Vector DB;
запрет на использование двух активных версий одновременно.

Релиз блокируется: да для критичных документов.

R-09. Потеря сообщения RabbitMQ

Меры:

persistent messages;
publisher confirms;
retry;
DLQ;
transactional outbox;
проверка восстановления после недоступности брокера;
алерт на необработанные документы.

Релиз блокируется: да.

R-10. Дубли при повторной доставке

Меры:

идемпотентный consumer;
уникальный eventId;
уникальное ограничение на версию документа;
повторная отправка одного события;
конкурентная обработка одинаковых сообщений;
проверка количества chunks после повторной доставки.

Релиз блокируется: да, если дубли влияют на ответы.

R-11. Ложный успешный статус

Меры:

статус изменяется только после подтверждения записи в Vector DB;
ошибки отдельных этапов сохраняются;
проверка согласованности PostgreSQL и Vector DB;
отдельные статусы RECEIVED, PROCESSING, INDEXED, FAILED;
запрет на код 200 при фактической ошибке процесса.

Релиз блокируется: да.

R-13. Деградация после смены модели или prompt

Меры:

версионирование модели и prompt;
обязательный evaluation-прогон перед релизом;
сравнение результатов с предыдущей версией;
контроль ключевых метрик;
canary release;
возможность быстрого rollback.

Релиз блокируется: да при выходе метрик за установленный порог.

R-18. Нерелевантные документы в RAG

Меры:

набор запросов с похожими терминами и разным смыслом;
оценка retrieval precision;
проверка top-k результатов;
тестирование фильтров по типу, версии и доступу;
ручной анализ пограничных запросов;
контроль качества разбиения документов на chunks.

Релиз блокируется: да для P0-сценариев.

5. Приоритет тестирования
В первую очередь

Проверяются риски с баллом от 16:

R-02 — неверный AI-ответ;
R-03 — ответ без источников;
R-04 — выдуманный ответ при отсутствии данных;
R-13 — деградация после смены модели;
R-18 — нерелевантный поиск RAG.
Во вторую очередь

Проверяются риски с баллом 10–15:

права доступа;
prompt injection;
потеря сообщений;
дубли;
версии документов;
ложные статусы;
утечки в логах;
потеря контекста.
После стабилизации критических потоков

Проверяются риски уровня Medium и Low:

удобство диагностики;
обработка больших документов;
редкие ошибки интерфейса;
дополнительные пользовательские настройки.
6. Остаточные риски

Не все риски AI-продукта можно устранить полностью.

Допустимыми могут быть признаны:

небольшая вариативность формулировок;
некритичное изменение порядка источников;
увеличение времени ответа в пределах установленного SLA;
отказ от ответа на неоднозначный вопрос;
ограниченная деградация в P2-сценариях.

Остаточный риск должен быть:

описан;
подтверждён результатами тестирования;
понятен Product Owner;
принят до выпуска релиза;
включён в post-release monitoring.
7. Решения QA Lead

QA Lead:

пересматривает риски после изменения архитектуры, модели или prompt;
определяет обязательный объём регрессии;
назначает владельцев проверок;
контролирует покрытие критических рисков;
выносит риски на Go / No-Go;
не рекомендует релиз при нарушении безопасности, потере данных или деградации критических AI-метрик.

Матрица рисков является живым документом и обновляется после:

появления нового дефекта;
изменения архитектуры;
смены LLM;
изменения системного prompt;
подключения нового источника данных;
инцидента в production.
