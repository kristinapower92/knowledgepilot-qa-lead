# Описание продукта и критических пользовательских потоков

## 1. О продукте

**KnowledgePilot** — корпоративный AI-ассистент, который помогает сотрудникам находить информацию во внутренних документах и получать ответы на естественном языке.

Продукт использует:

- LLM для формирования ответа;
- RAG для поиска релевантных документов и фрагментов;
- PostgreSQL для хранения пользователей, диалогов и статусов;
- векторную базу данных для поиска по документам;
- RabbitMQ для асинхронной индексации;
- REST API для взаимодействия между клиентом и backend;
- систему логирования и аудита для анализа запросов и ошибок.

## 2. Пользователи системы

### Сотрудник

Может:

- задавать вопросы;
- получать ответы со ссылками на источники;
- продолжать диалог с учётом контекста;
- просматривать историю запросов;
- загружать документы при наличии прав.

### Владелец документа

Может:

- загружать новую версию документа;
- запускать переиндексацию;
- отслеживать статус обработки;
- ограничивать доступ к документу.

### Администратор

Может:

- управлять ролями и правами доступа;
- просматривать технические статусы;
- анализировать ошибки индексации;
- повторно запускать обработку документов;
- контролировать состояние очередей.

## 3. Основные компоненты

| Компонент | Назначение |
|---|---|
| Web Interface | Отправка запросов и отображение ответов |
| Assistant API | Авторизация, валидация и обработка запросов |
| LLM Orchestrator | Управление prompt, контекстом и вызовами модели |
| RAG Service | Поиск релевантных документов и фрагментов |
| Vector Database | Хранение векторных представлений документов |
| PostgreSQL | Пользователи, роли, диалоги и статусы |
| RabbitMQ | Передача сообщений на индексацию |
| Indexing Service | Разбор, подготовка и индексация документов |
| Audit Service | Логи, события и трассировка запросов |

## 4. Критические пользовательские потоки

### Поток 1. Получение ответа по внутреннему документу

1. Пользователь отправляет вопрос.
2. Система проверяет авторизацию и права доступа.
3. RAG Service ищет релевантные документы.
4. LLM формирует ответ на основе найденного контекста.
5. Пользователь получает ответ и ссылки на источники.

#### Основные риски

- ответ не соответствует источникам;
- AI добавляет информацию, которой нет в документах;
- пользователю показывается закрытый документ;
- источники не относятся к ответу;
- при отсутствии информации система придумывает ответ;
- ошибка LLM маскируется под успешный результат.

---

### Поток 2. Продолжение диалога

1. Пользователь задаёт первый вопрос.
2. Система сохраняет контекст.
3. Пользователь задаёт уточняющий вопрос.
4. Ответ формируется с учётом предыдущих сообщений.

#### Основные риски

- теряется контекст;
- контекст одного пользователя попадает другому;
- старые сообщения искажают новый ответ;
- слишком длинный диалог приводит к обрезанию важной информации;
- система использует устаревшие данные.

---

### Поток 3. Загрузка и индексация документа

1. Пользователь загружает документ.
2. Backend сохраняет метаданные.
3. В RabbitMQ публикуется сообщение.
4. Indexing Service получает сообщение.
5. Документ разбивается на фрагменты.
6. Фрагменты сохраняются в векторную базу.
7. Статус документа меняется на `INDEXED`.
8. Пользователь получает уведомление.

#### Основные риски

- сообщение теряется при недоступности брокера;
- повторная доставка создаёт дубликаты;
- документ получает статус `INDEXED`, хотя обработка завершилась ошибкой;
- новая версия документа не заменяет старую;
- невалидный файл ломает consumer;
- отсутствуют retry и DLQ;
- пользователю показывается устаревшая версия документа.

---

### Поток 4. Обновление документа

1. Владелец загружает новую версию.
2. Система сохраняет версию и запускает переиндексацию.
3. Старые фрагменты удаляются или помечаются неактуальными.
4. Новые данные становятся доступны для поиска.

#### Основные риски

- одновременно используются старая и новая версии;
- старые фрагменты остаются в поиске;
- во время переиндексации документ становится недоступен;
- обновление одного документа влияет на другие;
- статус не соответствует фактическому состоянию.

---

### Поток 5. Разграничение доступа

1. Администратор назначает пользователю роль.
2. Пользователь задаёт вопрос.
3. Поиск выполняется только по разрешённым документам.
4. Источники фильтруются по правам доступа.

#### Основные риски

- пользователь получает ответ на основе закрытого документа;
- ссылка скрыта, но информация из документа попадает в ответ;
- права обновились, но кэш продолжает использовать старые данные;
- роль можно изменить через API без необходимых полномочий.

## 5. Нефункциональные требования

### Производительность

- пользователь должен получить первый ответ в согласованное время;
- длительная работа LLM не должна блокировать интерфейс;
- индексация одного большого документа не должна останавливать обработку остальных.

### Надёжность

- сообщения не должны теряться;
- повторная обработка должна быть идемпотентной;
- система должна восстанавливаться после недоступности брокера или LLM-провайдера.

### Безопасность

- пользователь видит только разрешённые документы;
- чувствительные данные не попадают в логи;
- prompt injection не должен обходить системные ограничения;
- системный prompt и внутренние настройки не раскрываются.

### Наблюдаемость

Для каждого запроса должны быть доступны:

- correlation ID;
- пользователь;
- версия модели;
- версия prompt;
- найденные источники;
- длительность этапов;
- код и причина ошибки.

## 6. Что является критичным для релиза

Релиз блокируется, если:

- пользователь получает доступ к закрытым данным;
- AI формирует опасный или заведомо недостоверный ответ;
- ответ не подтверждается источниками в критических сценариях;
- сообщение об индексации теряется;
- повторная обработка создаёт дубликаты;
- документ имеет успешный статус при фактической ошибке;
- отсутствует возможность проследить запрос по логам;
- критический пользовательский поток не работает.
