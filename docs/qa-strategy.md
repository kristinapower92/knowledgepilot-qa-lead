# QA-стратегия KnowledgePilot

## 1. Цель стратегии

Цель QA-стратегии — обеспечить предсказуемое качество AI/LLM-продукта, снизить риски утечки данных, галлюцинаций, ошибок в источниках и потери документов при асинхронной обработке.

Качество KnowledgePilot оценивается не только по тому, работает ли интерфейс и возвращает ли API код `200`.

Для продукта критично:

- насколько ответ соответствует внутренним документам;
- можно ли проверить ответ по источникам;
- не получает ли пользователь доступ к закрытой информации;
- корректно ли система ведёт себя при отсутствии данных;
- сохраняются ли сообщения и документы при сбоях;
- можно ли восстановить цепочку обработки по логам;
- остаётся ли поведение продукта приемлемым после смены модели, prompt или базы знаний.

---

## 2. Принципы качества

### Тестирование на основе рисков

Приоритет проверки определяется возможным ущербом для пользователя и бизнеса.

Наиболее опасные области:

1. нарушение прав доступа;
2. уверенный, но недостоверный AI-ответ;
3. ответ без подтверждения источниками;
4. потеря документа или сообщения;
5. некорректный статус индексации;
6. отсутствие наблюдаемости;
7. деградация качества после обновления модели или prompt.

### Проверяем не только результат, но и путь его получения

Для AI-ответа важно проверить:

- какие документы были найдены;
- какие фрагменты переданы в LLM;
- была ли соблюдена фильтрация по правам;
- какая версия модели использовалась;
- какая версия prompt применялась;
- почему система решила сформировать ответ или отказаться от него.

### AI-качество не сводится к проверке точного текста

Одинаковый запрос может получать разные формулировки ответа.

Поэтому оцениваются:

- фактическая корректность;
- полнота;
- релевантность;
- подтверждённость источниками;
- безопасность;
- соблюдение ограничений;
- стабильность смысла при повторных запросах.

### Автоматизация дополняет экспертную оценку

Автоматизируются стабильные и формализуемые проверки:

- API-контракты;
- JSON Schema;
- обязательные поля;
- права доступа;
- статусы обработки;
- события RabbitMQ;
- наличие источников;
- структура AI-ответа;
- regression evaluation dataset.

Ручными остаются:

- exploratory testing;
- оценка сложных AI-ответов;
- анализ неоднозначных запросов;
- UX;
- новые виды prompt injection;
- проверка качества после значительных изменений модели.

## 3. Уровни тестирования

| Уровень | Основной фокус | Ответственные |
|---|---|---|
| Unit | Бизнес-правила, парсинг документов, фильтрация данных | Разработчики |
| Component | Работа отдельных сервисов в изоляции | Разработчики и QA |
| Contract | API-контракты и схемы событий | QA и разработчики |
| Integration | LLM, RAG, PostgreSQL, RabbitMQ, Vector DB | QA |
| End-to-End | Критические пользовательские потоки | QA |
| AI Evaluation | Качество, безопасность и устойчивость ответов | QA и продукт |
| UAT | Соответствие бизнес-сценариям | Product Owner и QA |

---

## 4. Виды тестирования

### Функциональное тестирование

Проверяются:

- отправка запроса;
- получение ответа;
- сохранение истории;
- работа с контекстом;
- загрузка документа;
- индексация;
- обновление версии;
- отображение статусов;
- управление ролями.

### API-тестирование

Проверяются:

- обязательные поля;
- типы данных;
- схемы JSON;
- HTTP-коды;
- обработка ошибок;
- авторизация;
- роли;
- correlation ID;
- таймауты;
- повторные запросы;
- идемпотентность.

### Интеграционное тестирование

Проверяются взаимодействия:

- Assistant API → LLM Orchestrator;
- LLM Orchestrator → LLM Provider;
- LLM Orchestrator → RAG Service;
- RAG Service → Vector Database;
- Backend → PostgreSQL;
- Backend → RabbitMQ;
- RabbitMQ → Indexing Service.

### Тестирование RAG

Проверяются:

- релевантность найденных документов;
- качество найденных фрагментов;
- корректность источников;
- отсутствие закрытых документов в выдаче;
- работа при противоречивых данных;
- реакция на отсутствие информации;
- актуальность результатов после переиндексации.

### Тестирование LLM

Проверяются:

- фактическая корректность;
- отсутствие галлюцинаций;
- соблюдение системных ограничений;
- обработка неоднозначных запросов;
- отказ от ответа при недостатке данных;
- устойчивость к переформулировкам;
- сохранение контекста;
- prompt injection;
- раскрытие системного prompt;
- небезопасный контент.

### Тестирование асинхронных процессов

Проверяются:

- доставка сообщения;
- повторная доставка;
- идемпотентность;
- retry;
- DLQ;
- восстановление после сбоя;
- поведение при недоступности брокера;
- соответствие статуса фактической обработке.

### Тестирование безопасности

Проверяются:

- права доступа;
- изоляция пользовательских данных;
- доступ к документам;
- утечки через AI-ответ;
- чувствительные данные в логах;
- попытки обхода ограничений через prompt injection;
- изменение ролей через API.

### Нефункциональное тестирование

Проверяются:

- время ответа;
- стабильность при параллельных запросах;
- обработка больших документов;
- деградация при недоступности LLM;
- восстановление после сбоев;
- наблюдаемость;
- качество логирования.

## 5. Приоритеты тестирования

### P0 — критические сценарии

- пользователь получает ответ только по разрешённым документам;
- закрытая информация не попадает в ответ;
- ответ подтверждается источниками;
- при отсутствии данных AI не придумывает ответ;
- документ успешно проходит полный цикл индексации;
- сообщения не теряются;
- повторная доставка не создаёт дубликаты;
- критические ошибки можно проследить по correlation ID.

### P1 — важные сценарии

- сохранение контекста;
- обновление документов;
- обработка противоречивых источников;
- таймаут LLM;
- retry и DLQ;
- история запросов;
- повторный запуск индексации.

### P2 — дополнительные сценарии

- фильтрация истории;
- необязательные параметры;
- косметические ошибки;
- редкие пользовательские настройки.

---

## 6. Стратегия тестовых данных

Для тестирования используются только синтетические данные.

Наборы данных включают:

- общедоступные документы;
- документы с ограниченным доступом;
- документы разных версий;
- противоречивые документы;
- устаревшие документы;
- документы без ответа на запрос;
- документы с одинаковыми терминами, но разным смыслом;
- невалидные и повреждённые файлы;
- большие документы;
- запросы с prompt injection.

Для AI evaluation создаётся отдельный набор:

- вопрос;
- ожидаемые факты;
- допустимые источники;
- запрещённые утверждения;
- критерии успешного ответа;
- уровень критичности.

---

## 7. Стратегия автоматизации

Автоматизация строится по принципу максимальной пользы для регрессии.

### Автоматизируются в первую очередь

- API smoke;
- авторизация и роли;
- API-контракты;
- схемы RabbitMQ-событий;
- статусы документов;
- идемпотентность;
- проверки доступа;
- наличие и формат источников;
- evaluation-набор стабильных AI-сценариев;
- post-deploy smoke.

### Не автоматизируются только ради процента покрытия

Не является целью автоматизировать:

- все формулировки AI-ответов;
- одноразовые проверки;
- нестабильные сценарии без понятного ожидаемого результата;
- редкие UI-детали с высокой стоимостью поддержки.

---

## 8. Работа с дефектами

### Severity

| Уровень | Пример |
|---|---|
| Blocker | Основной пользовательский поток полностью недоступен |
| Critical | Утечка закрытых данных, потеря документов, опасный AI-ответ |
| Major | Неверный источник, некорректный статус, потеря контекста |
| Minor | Частичная ошибка отображения или некритичное неудобство |
| Trivial | Косметический дефект |

### Дефект блокирует релиз, если

- нарушена безопасность данных;
- AI выдаёт опасную или недостоверную информацию в критическом сценарии;
- отсутствует подтверждение ответа источниками;
- сообщение или документ теряется;
- успешный статус не соответствует фактическому результату;
- не работает критический пользовательский поток;
- невозможно диагностировать ошибку.

---

## 9. Quality Gates

Релиз может быть рекомендован, если:

- открытых `Blocker` и `Critical` нет;
- P0-сценарии пройдены на 100%;
- P1-сценарии пройдены не менее чем на 95%;
- API-контракты и схемы событий прошли проверки;
- security-сценарии пройдены;
- retry и DLQ проверены;
- AI evaluation соответствует согласованным порогам;
- проверена актуальность источников;
- настроены логи, метрики и алерты;
- выполнен smoke на Stage;
- подготовлен rollback;
- остаточные риски зафиксированы и приняты Product Owner.

## 10. Роли и ответственность

| Роль | Ответственность |
|---|---|
| QA Lead | Стратегия, риски, планирование, приоритеты, метрики, Go / No-Go |
| QA Engineer 1 | API, PostgreSQL, авторизация и интеграции |
| QA Engineer 2 | LLM, RAG, evaluation dataset и AI safety |
| QA Engineer 3 | RabbitMQ, индексация, E2E и отказоустойчивость |
| Developers | Unit/component tests, исправления, техническая поддержка |
| Product Owner | Бизнес-критерии и принятие остаточных рисков |
| Analyst | Требования, сценарии и правила доступа |

QA Lead не только распределяет задачи, но и участвует в технически сложных проверках, анализе дефектов и ревью тестовых артефактов.

## 11. Коммуникации и контроль качества

### До разработки

QA участвует в refinement:

- анализирует требования;
- задаёт вопросы по серым зонам;
- определяет риски;
- проверяет тестируемость;
- согласует логи, метрики и ошибки;
- уточняет критерии приёмки.

### Во время разработки

- QA синхронизируется с разработчиками и аналитиком;
- критические изменения проходят cross-review;
- риски фиксируются в risk log;
- дефекты обсуждаются на triage;
- изменения API и событий проверяются до интеграции.

### Перед релизом

QA Lead:

- анализирует результаты прогонов;
- проверяет открытые дефекты;
- оценивает метрики;
- фиксирует остаточные риски;
- готовит тестовый отчёт;
- даёт рекомендацию `Go / No-Go`.

### После релиза

- выполняется production smoke;
- контролируются ошибки API;
- проверяется глубина очередей и DLQ;
- анализируется качество AI-ответов;
- проводится разбор проблем релиза;
- обновляется regression-набор.
