## 2. Цель тестирования

Цель тестирования — подтвердить готовность релиза, в котором реализованы:

- фильтрация RAG-источников по правам доступа;
- отказ AI от ответа при недостатке данных;
- отображение подтверждающих источников;
- переиндексация новой версии документа;
- retry и DLQ для ошибок индексации;
- защита от потери и дублирования сообщений;
- расширенное логирование AI-запросов.

### Критические дефекты

В ходе первого прогона обнаружены два дефекта уровня `Critical`.

#### BUG-001

**AI раскрывает информацию из закрытого документа в тексте ответа.**

Пользователь с базовой ролью получал данные из документа `management_only`, хотя источник не отображался в API и интерфейсе.

Связанный риск:

```text
R-01 — доступ к закрытой информации
```

#### BUG-002

**Документ теряется при недоступности RabbitMQ, несмотря на успешный ответ API.**

API возвращал `202 Accepted`, но событие индексации не сохранялось и не публиковалось после восстановления брокера.

Связанный риск:

```text
R-09 — потеря сообщения об индексации
```

## 6. Первоначальное решение QA Lead

По итогам первого прогона была дана рекомендация:

```text
NO-GO
```

### Причины

- подтверждена утечка закрытой информации;
- нарушена ролевая модель;
- документ мог быть потерян после успешного ответа API;
- два P0-сценария не пройдены;
- существовал риск использования устаревшей базы знаний;
- безопасного временного обходного решения не было.

Релиз был заблокирован до исправления критических дефектов.

---

## 7. Выполненные исправления

### Исправление BUG-001

Командой разработки были внесены изменения:

- фильтрация прав перенесена до передачи chunks в LLM;
- access policy применяется на этапе retrieval;
- retrieval-кэш учитывает пользователя и его роли;
- смена прав приводит к инвалидации кэша;
- в audit-логи добавлен результат проверки доступа;
- закрытые документы исключаются из контекста модели.

### Исправление BUG-002

Для надёжной доставки событий реализованы:

- transactional outbox;
- фоновая публикация сохранённых событий;
- publisher confirms;
- уникальный `eventId`;
- идемпотентность consumer;
- контроль документов в промежуточном статусе;
- алерт на превышение SLA;
- сохранение причины ошибки.

## 12. Остаточные риски

### Вариативность AI-ответов

Формулировки могут отличаться между запусками при сохранении одинакового смысла.

**Контроль:**

- production monitoring;
- пользовательская обратная связь;
- регулярный evaluation-прогон.

### Зависимость от LLM Provider

При деградации внешнего провайдера возможно увеличение времени ответа.

**Контроль:**

- timeout;
- fallback;
- мониторинг latency;
- алерт на рост ошибок.

### Новые виды prompt injection

Существующий набор атак не гарантирует защиту от всех будущих вариантов.

**Контроль:**

- регулярное обновление security dataset;
- анализ инцидентов;
- повторные security-review.

### Большие документы

Документы предельного размера могут обрабатываться дольше среднего SLA.

**Контроль:**

- мониторинг очереди;
- ограничение размера;
- отдельный алерт на длительную обработку.
