# BUG-002 — документ теряется при недоступности RabbitMQ, несмотря на успешный ответ API

## Основная информация

| Поле | Значение |
|---|---|
| Severity | Critical |
| Priority | High |
| Компоненты | Assistant API, RabbitMQ, Indexing Service |
| Окружение | Stage |
| Версия приложения | 1.3.0-rc2 |
| Связанный риск | R-09 — потеря сообщения об индексации |
| Статус | Open |

---

## Описание

При недоступности RabbitMQ API загрузки документа возвращает успешный ответ, хотя событие индексации не публикуется.

После восстановления брокера:

- сообщение не появляется в основной очереди;
- повторная отправка не выполняется;
- сообщение отсутствует в DLQ;
- документ остаётся в промежуточном статусе;
- автоматическое восстановление обработки не происходит.

Пользователь получает подтверждение загрузки, но документ фактически не попадает в базу знаний.

---

## Предусловия

1. Сервис KnowledgePilot доступен.
2. Пользователь авторизован с правом загрузки документов.
3. RabbitMQ недоступен.
4. Основная очередь индексации и DLQ очищены.
5. В системе отсутствует документ с тестовым идентификатором.

---

## Тестовые данные

```text
File: remote-work-policy-v3.pdf
Document title: Политика удалённой работы
Version: 3.0
Access: employees
```

---

## Шаги воспроизведения

1. Остановить RabbitMQ либо заблокировать соединение приложения с брокером.
2. Отправить запрос:

```http
POST /api/v1/documents
Authorization: Bearer <valid_token>
Content-Type: multipart/form-data
X-Correlation-ID: 8abf1229-8e7a-47d7-b8f2-316bd5311d11
```

3. Передать валидный файл и метаданные:

```json
{
  "title": "Политика удалённой работы",
  "version": "3.0",
  "accessGroup": "employees"
}
```

4. Проверить HTTP-ответ.
5. Проверить запись документа в PostgreSQL.
6. Проверить основную очередь индексации.
7. Проверить DLQ.
8. Восстановить RabbitMQ.
9. Подождать пять минут.
10. Повторно проверить очередь, DLQ и статус документа.

---

## Фактический результат

API возвращает:

```http
HTTP/1.1 202 Accepted
```

```json
{
  "documentId": "doc-49821",
  "status": "RECEIVED",
  "message": "Document accepted for indexing"
}
```

При этом:

- событие `document.indexing.requested` не опубликовано;
- сообщение отсутствует в основной очереди;
- сообщение отсутствует в DLQ;
- повторная публикация после восстановления RabbitMQ не выполняется;
- документ остаётся в статусе `RECEIVED`;
- запись о причине ошибки пользователю и администратору не показывается.

Через пять минут статус не изменяется:

```json
{
  "documentId": "doc-49821",
  "status": "RECEIVED"
}
```

Документ отсутствует в Vector Database и не участвует в поиске.

---

## Ожидаемый результат

Система не должна подтверждать успешную передачу документа на индексацию, если событие не было надёжно сохранено.

Допустимые варианты поведения:

### Вариант 1. Transactional outbox

1. Документ и событие сохраняются в одной транзакции.
2. API возвращает успешный ответ.
3. После восстановления RabbitMQ событие автоматически публикуется.
4. Документ проходит индексацию.

### Вариант 2. Контролируемая ошибка

Если событие сохранить невозможно:

```http
HTTP/1.1 503 Service Unavailable
```

```json
{
  "code": "MESSAGE_BROKER_UNAVAILABLE",
  "message": "Document indexing is temporarily unavailable",
  "correlationId": "8abf1229-8e7a-47d7-b8f2-316bd5311d11"
}
```

При этом система не должна сообщать, что документ принят на индексацию.

---

## Влияние

Дефект нарушает основной пользовательский поток:

```text
Загрузка документа → индексация → появление в базе знаний
```

Возможные последствия:

- пользователь считает загрузку успешной;
- документ не появляется в поиске;
- AI использует устаревшую базу знаний;
- владелец документа не получает понятного статуса ошибки;
- повторная загрузка может создать лишние записи;
- команда поддержки не может быстро определить причину;
- потерянные документы приходится искать вручную.

Особенно критично это для документов, содержащих:

- новые внутренние правила;
- изменения льгот;
- требования информационной безопасности;
- обязательные процедуры;
- обновлённые регламенты.

---

## Частота воспроизведения

```text
5 из 5 запусков
```

Во всех проверках API возвращал `202 Accepted`, но событие не появлялось после восстановления RabbitMQ.

---

## Дополнительные проверки

### RabbitMQ доступен

При доступном брокере сценарий работает корректно:

- возвращается `202 Accepted`;
- событие появляется в очереди;
- Indexing Service обрабатывает документ;
- статус меняется на `INDEXED`;
- документ появляется в Vector Database.

### RabbitMQ недоступен кратковременно

Если восстановить RabbitMQ через 30 секунд:

- потерянное событие всё равно не публикуется;
- retry отсутствует;
- статус остаётся `RECEIVED`.

### Повторная загрузка того же файла

Повторная загрузка создаёт новый `documentId`.

Это может привести к накоплению записей в статусе `RECEIVED`.

### Correlation ID

```text
8abf1229-8e7a-47d7-b8f2-316bd5311d11
```

### Лог Assistant API

```text

```

После ошибки приложение продолжает формирование успешного HTTP-ответа.

### PostgreSQL

Запись создана:

```text

version: 3.0

```

Поле с причиной ошибки отсутствует.

### RabbitMQ

```text
Main queue: 0 messages
Retry queue: 0 messages
DLQ: 0 messages
```

## Предполагаемая зона проблемы

Возможные причины:

- сохранение документа и публикация события выполняются независимо;
- ошибка publisher перехватывается, но не влияет на HTTP-ответ;
- отсутствует transactional outbox;
- publisher confirms не используются;
- отсутствует механизм фоновой повторной публикации;
- статус `RECEIVED` не имеет SLA-контроля;
- не настроен алерт на документы, зависшие в промежуточном статусе.

Предположение требует подтверждения командой разработки.

---

## Критерии исправления

Исправление считается корректным, если:

- успешный ответ возвращается только после надёжной фиксации события;
- временная недоступность RabbitMQ не приводит к потере документа;
- после восстановления брокера сохранённое событие автоматически публикуется;
- повторная публикация не создаёт дубликаты;
- используется уникальный `eventId`;
- документ получает понятный статус ошибки при невозможности продолжить обработку;
- причина сбоя сохраняется;
- доступен correlation ID;
- настроен мониторинг документов, находящихся в промежуточном статусе дольше SLA;
- пользователь не получает ложное сообщение об успешной индексации.

---

## Объём регрессии

После исправления необходимо проверить:

- загрузку документа при доступном RabbitMQ;
- недоступность RabbitMQ до отправки запроса;
- отключение RabbitMQ во время публикации;
- восстановление брокера;
- повторную публикацию сохранённого события;
- publisher confirms;
- retry;
- DLQ;
- transactional outbox;
- идемпотентность consumer;
- повторную доставку одного `eventId`;
- перезапуск Assistant API;
- перезапуск Indexing Service;
- перезапуск RabbitMQ;
- соответствие статусов в PostgreSQL;
- отсутствие дублей в Vector Database;
- логирование и correlation ID;
- алерты на зависшие документы.

---

## Решение QA Lead

Дефект блокирует релиз `1.3.0`.

Рекомендация:

```text
NO-GO
```

Причина:

- пользователь получает ложный успешный результат;
- сообщение и документ фактически теряются;
- автоматическое восстановление отсутствует;
- дефект воспроизводится стабильно;
- нарушается критический поток индексации;
- отсутствует безопасное временное решение для production.

Повторное решение `Go / No-Go` возможно только после:

1. исправления механизма доставки;
2. проверки отказоустойчивости;
3. полного P0-прогона;
4. подтверждения отсутствия потери и дублирования событий.
