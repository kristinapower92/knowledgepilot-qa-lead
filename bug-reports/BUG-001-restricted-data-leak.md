# BUG-001 — AI раскрывает информацию из закрытого документа в тексте ответа

## Основная информация

| Поле | Значение |
|---|---|
| Severity | Critical |
| Priority | High |
| Компоненты | RAG Service, Access Control, LLM Orchestrator |
| Окружение | Stage |
| Версия приложения | 1.3.0-rc2 |
| Версия модели | llm-enterprise-4.1 |
| Версия system prompt | 1.3.7 |
| Связанный риск | R-01 — доступ к закрытой информации |
| Статус | Open |

---

## Описание

Пользователь с базовой ролью получает в тексте AI-ответа информацию из документа с уровнем доступа `management_only`.

Закрытый документ не отображается в списке источников, однако его содержимое используется при формировании ответа.

Таким образом, фильтрация применяется только к отображению источников, но не предотвращает передачу закрытого контекста в LLM.

---

## Предусловия

1. В системе существует документ:

```text
Document ID: COMPENSATION-LEADS-001
Title: Бонусная политика руководителей
Version: 3.1
Access: management_only
Status: active
```

2. Документ содержит информацию:

```text
Целевой годовой бонус руководителя составляет
20% от годового фиксированного дохода.
```

3. Пользователь авторизован с ролью:

```text
employee
```

4. У пользователя отсутствует доступ к документам категории `management_only`.

---

## Шаги воспроизведения

1. Авторизоваться под пользователем с ролью `employee`.
2. Открыть новый диалог.
3. Отправить запрос:

```text
Какой размер годового бонуса установлен для руководителей?
```

4. Дождаться ответа.
5. Проверить текст ответа.
6. Проверить массив `sources` в API-ответе.
7. Найти запрос в логах по `correlationId`.
8. Проверить документы и chunks, переданные из RAG Service в LLM Orchestrator.

---

## Фактический результат

AI возвращает ответ:

```text
Целевой годовой бонус руководителя составляет
20% от годового фиксированного дохода.
```

При этом:

```json
{
  "sources": []
}
```

В технических логах RAG Service присутствует chunk из документа:

```text
COMPENSATION-LEADS-001
```

Документ был передан в LLM, несмотря на отсутствие у пользователя необходимых прав.

---

## Ожидаемый результат

Документы должны фильтроваться по правам доступа **до передачи контекста в LLM**.

Для пользователя с ролью `employee`:

- закрытый документ не участвует в retrieval;
- его chunks не передаются в LLM;
- информация из документа отсутствует в ответе;
- документ отсутствует в `sources`;
- попытка доступа фиксируется в audit-логах.

Допустимый ответ:

```text
В доступных вам документах недостаточно информации
для точного ответа.
```

---

## Влияние

Дефект приводит к утечке конфиденциальной корпоративной информации.

Пользователь может получить закрытые данные, даже если:

- источник не отображается в интерфейсе;
- ссылка на документ скрыта;
- прямой доступ к документу запрещён;
- API источников возвращает пустой массив.

Скрытие источника не предотвращает раскрытие его содержимого через сгенерированный ответ.

---

## Частота воспроизведения

```text
5 из 5 запусков
```

Дефект стабильно воспроизводится при новых диалогах и разных формулировках запроса.

Примеры:

```text
Какой бонус получают руководители?
```

```text
Как рассчитывается годовая премия менеджмента?
```

```text
Сколько процентов составляет бонус руководителя?
```

---

## Дополнительные наблюдения

При запросе от пользователя с ролью `manager` система работает корректно:

- документ участвует в retrieval;
- ответ содержит правильную информацию;
- источник отображается;
- версия документа указана корректно.

Проблема проявляется только при недостаточном уровне доступа.

---

## Предполагаемая зона проблемы

Возможные причины:

- фильтрация прав применяется после retrieval;
- список источников фильтруется отдельно от контекста LLM;
- векторный поиск не получает ограничения по ролям;
- кэш retrieval не учитывает пользователя и его права;
- LLM Orchestrator получает chunks до проверки access policy.

Предположение требует подтверждения командой разработки.

---

## Критерии исправления

Исправление считается корректным, если:

- access-фильтрация выполняется до передачи chunks в LLM;
- закрытый документ не появляется в retrieval-результатах пользователя;
- информация из закрытого документа отсутствует в ответе;
- кэш учитывает пользователя, роль и права доступа;
- смена роли инвалидирует ранее доступный контекст;
- попытка обращения к закрытым данным фиксируется в audit-логах;
- разрешённые документы продолжают обрабатываться без регрессии.

---

## Объём регрессии

После исправления необходимо проверить:

- запросы по доступным документам;
- запросы по закрытым документам;
- документы с несколькими группами доступа;
- изменение роли пользователя;
- отзыв ранее выданного доступа;
- кэшированные retrieval-результаты;
- продолжение старого диалога после изменения прав;
- параллельные сессии разных пользователей;
- прямой запрос через API;
- источники и текст ответа;
- audit-логи;
- prompt injection с просьбой раскрыть закрытые данные.

---

## Решение QA Lead

Дефект блокирует релиз `1.3.0`.

Рекомендация:

```text
NO-GO
```

Причина:

- нарушена модель доступа;
- подтверждена утечка закрытой информации;
- риск относится к категории Critical;
- обход воспроизводится стабильно;
- безопасного временного решения для production нет.
