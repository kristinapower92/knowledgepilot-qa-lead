# KnowledgePilot — QA Lead Portfolio Project

### QA-стратегия и тестирование AI/LLM-продукта

KnowledgePilot — синтетический проект корпоративного AI-ассистента, который отвечает на вопросы сотрудников на основе внутренней базы знаний, находит релевантные документы и указывает источники ответа.

Проект демонстрирует мой подход к работе **QA Lead** в продукте, объединяющем LLM, RAG, REST API, базы данных и асинхронные интеграции.

> Все названия, данные, архитектура и дефекты в проекте вымышлены.  
> Проект не содержит информации моих работодателей и создан специально для портфолио.

## О продукте

KnowledgePilot помогает сотрудникам компании:

- задавать вопросы на естественном языке;
- находить информацию во внутренних документах;
- получать краткий ответ со ссылками на источники;
- сохранять контекст диалога;
- загружать документы для последующей индексации;
- получать уведомление после завершения обработки документа.

Пример пользовательского запроса:

> Сколько дополнительных оплачиваемых выходных положено сотруднику после трёх лет работы и в каком документе это указано?

Система должна найти соответствующий внутренний документ, сформировать ответ и показать использованные источники.

## Основные зоны тестирования

### AI и LLM

- релевантность и полнота ответа;
- галлюцинации;
- фактическая корректность;
- вариативность ответов;
- соблюдение системных ограничений;
- обработка неоднозначных запросов;
- сохранение контекста диалога;
- prompt injection;
- раскрытие системного промпта;
- небезопасный или запрещённый контент.

### RAG

- корректность поиска документов;
- релевантность найденных фрагментов;
- соответствие ответа источникам;
- корректность ссылок и цитат;
- обработка противоречивых документов;
- отсутствие ответа при недостатке информации;
- обновление результатов после переиндексации;
- разграничение доступа к документам.

### API и backend

- REST API;
- авторизация и роли;
- схемы JSON;
- обязательные поля и типы данных;
- коды и тела ошибок;
- PostgreSQL;
- логирование и correlation ID;
- таймауты и повторные запросы;
- интеграция с внешним LLM-провайдером.

### Асинхронная индексация

- доставка сообщений через RabbitMQ;
- повторная обработка;
- идемпотентность;
- retry;
- DLQ;
- потеря сообщения при недоступности брокера;
- восстановление после сбоя;
- статусы обработки документа.

## Ключевые риски

| Риск | Влияние |
|---|---|
| AI формирует убедительный, но неверный ответ | Пользователь принимает ошибочное решение |
| Ответ не подтверждается источниками | Снижается доверие к продукту |
| Пользователь получает доступ к закрытому документу | Утечка корпоративных данных |
| Документ не переиндексируется после обновления | Система использует устаревшую информацию |
| Сообщение теряется при недоступности брокера | Загруженный документ не появляется в базе знаний |
| Повторная доставка создаёт дубликаты | Результаты поиска становятся нестабильными |
| Ошибка LLM возвращается пользователю как успешный ответ | Ложное ощущение корректной работы системы |

## Quality Gates

Релиз может быть рекомендован к запуску, если:

- нет открытых дефектов уровня `Blocker` и `Critical`;
- критические пользовательские сценарии пройдены на 100%;
- API-контракты соответствуют спецификации;
- не выявлено доступа к чужим или закрытым документам;
- проверены retry, DLQ и восстановление после сбоя;
- ответы подтверждаются релевантными источниками;
- AI не выдаёт ответ при отсутствии достаточной информации;
- пройден набор проверок на prompt injection;
- метрики качества находятся в согласованных пределах;
- остаточные риски зафиксированы и приняты владельцем продукта.

## Организация работы QA-команды

В синтетическом проекте предусмотрена команда из трёх QA-инженеров.

| Роль | Зона ответственности |
|---|---|
| QA Lead | Стратегия, риски, планирование, метрики и Go / No-Go |
| QA Engineer 1 | REST API, PostgreSQL, авторизация и интеграции |
| QA Engineer 2 | LLM, RAG, evaluation-наборы и безопасность |
| QA Engineer 3 | Очереди, индексация, E2E и отказоустойчивость |

Критические сценарии проходят перекрёстное ревью. QA участвует в refinement до начала разработки, чтобы заранее находить серые зоны и повышать тестируемость продукта.
